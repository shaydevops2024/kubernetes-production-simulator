Fix CrashLoopBackOff Error

Learn to diagnose and resolve pod crash issues in Kubernetes

Step 1 of 5
1

Identify the Failing Pod

$ kubectl get pods -n k8s-multi-demo

üîç What this does

Lists all pods in the k8s-multi-demo namespace, showing their current status, restarts count, and age.

‚úÖ Expected output

Look for a pod with STATUS showing "CrashLoopBackOff" and a high RESTARTS count.

üí°
Pro Tip

CrashLoopBackOff means the container keeps crashing. Kubernetes will keep trying to restart it with increasing delays.

2

Check Pod Logs

$ kubectl logs -n k8s-multi-demo -l app=k8s-demo-app --previous

üîç What this does

Retrieves logs from the previous container instance (before it crashed) to see the error messages.

‚úÖ Expected output

Error messages showing why the application crashed - could be missing config, connection errors, or code exceptions.

üí°
Pro Tip

The --previous flag is crucial here since the current container may have just started and has no useful logs yet.

3

Describe Pod Events

$ kubectl describe pod -n k8s-multi-demo -l app=k8s-demo-app | grep -A 20 "Events:"

üîç What this does

Shows detailed pod information including the Events section which tracks all actions Kubernetes has taken.

‚úÖ Expected output

Events showing "BackOff" warnings, container start/stop events, and potentially "Failed" messages with reasons.

4

Apply the Fix

$ kubectl set env deployment/k8s-demo-app -n k8s-multi-demo APP_CONFIG=fixed

üîç What this does

Updates the deployment with the correct environment variable, triggering a new rollout with the fixed configuration.

‚úÖ Expected output

"deployment.apps/k8s-demo-app env updated" - Kubernetes will now create new pods with the fix.

üí°
Pro Tip

This triggers a rolling update. Old pods are terminated only after new pods are healthy.

5

Verify the Fix

$ kubectl get pods -n k8s-multi-demo -w

üîç What this does

Watches pods in real-time (-w flag) so you can see the new pods starting up and becoming Ready.

‚úÖ Expected output

Pods should transition from "ContainerCreating" ‚Üí "Running" with STATUS "1/1" and RESTARTS staying at 0.

Success!

If pods stay in Running state without restarting, you've successfully fixed the CrashLoopBackOff issue!
